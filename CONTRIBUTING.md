# Contributing

This repo is one operator's field manual and a reproducible build path — not a software project in the traditional sense. Contributions that fit:

- **Corrections** — a wrong claim, a broken link, a typo. Open an issue or a PR directly.
- **Field notes** — you hit a platform-specific gap or debugged a real failure in one of the pillars (macOS/Linux equivalents of [Field Conditions: Windows](manual/02-field-conditions-windows.md)?). Open an issue with what broke, how you diagnosed it, and what fixed it. See [field-notes/](field-notes/README.md) for the format: symptom → eliminated hypotheses → root cause → fix → transferable lesson.
- **Upstream reports** — you followed the [loop](upstream/README.md) and sent a fix to one of the pillar projects because of something documented here? Open an issue; strong entries get linked from the ledger.
- **Showcase** — built something because you followed this thesis? Open an issue with what it does, what gap it closes, and a link.

What doesn't fit: swapping in a different "pillar 1/2/3" project as *the* canonical reference. The thesis is tool-agnostic (see the [FAQ](manual/03-faq.md)), but this repo's own build path documents the three projects the author actually uses and can vouch for.

## Style

Match the existing voice: concrete over hype, honest about gaps, field notes rather than marketing copy. If a claim can't be backed by something reproducible, say so plainly instead of smoothing it over.
